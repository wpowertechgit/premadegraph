\section{A korai változat}

Az eredeti rendszer jóval közelebb állt egy adatvezérelt játékosnézőhöz, mint a jelenlegi kutatási platformhoz. A hangsúly az volt, hogy a Riot API-ból érkező nyers adatok egy helyre kerüljenek, a játékosokhoz legalább alapvető pontszámok rendelődjenek, és a kapcsolatokból egyszerű gráfkimenet szülessen.

A korai pontozási logika (\textit{add\_new\_players.js}) egyszerű képletekkel dolgozott: a \textit{feedscore} a halálokat büntette a kill/assist mennyiségéhez képest, az \textit{opscore} öléseket, asszisztokat, aranyat és vision score-t vett figyelembe. Nyers volt, de megmutatta, hogy a puszta KDA-logikán túl is számolható részben kompozit mutató. Ebből nőtt ki a jelenlegi, szerepkörérzékeny rendszer.

A korai \textit{build\_graph.py} Python-központú volt: meccsekből kapcsolati gráfot épített, HTML és PyVis exportokat készített. A klaszterek és gráfnézetek csak exportként éltek, nem belső, lekérdezhető objektumokként. Ebből az időszakból maradt a \textit{fetch\_clusters.py}/\textit{assign\_countries.py}/\textit{conclude.py} ország-ág is, amely klaszternevekből próbált földrajzi következtetéseket levonni: módszertanilag bizonytalan vonal, de innen nőtt ki az a szemlélet, hogy a klasztereket értelmezési egységekként kezeljük. Ezen az úton váltak a klaszterprofilok végül a Tribal NeuroSim szimulációs bemenetének alapjává.

\section{A jelenlegi architektúra}

A projekt jelenlegi architektúrája hibrid. A fő komponensek: Python (adatgyűjtés, gráfépítés, legacy-exportok), SQLite (központi perzisztencia), Node.js/Express (API-héj és orchesztráció), Rust (útvonalkeresés, gráfanalitika, globális artifact-export), React/Vite (frontend és vizualizáció), C\# ASP.NET Core (különálló adatbázis-böngésző) és a Tribal NeuroSim szimulációs motor, amelynek bemenete a gráf-pipeline klaszterartifaktumaiból származik. A rendszer \textit{docker-compose.yml} segítségével konténerizált formában is futtatható: a backend és a frontend külön konténerben indul, a fejlesztői és prod-közelű futtatás egyaránt támogatott.

Az alábbi komponensdiagram a teljes rendszer magas szintű szerkezetét mutatja be: a Python adatgyűjtő és gráfépítő worker-eket, a Node.js/Express API réteget, a Rust analitikai motort, a React/Vite frontendet és a C\# ASP.NET Core adatbázis-böngészőt, valamint az ezeket összekötő kommunikációs csatornákat és tárolási egységeket. A diagram jól mutatja, hogy a frontend--Rust Pathfinder összeköttetés a Node.js API-n keresztül fut: a frontend közvetlenül nem hívja a Rust motort, hanem HTTP-végpontokon keresztül kommunikál a Node rétegrel, amely a Rust folyamatot stdin/stdout csatornán keresztül vezérli.

\section{Technológiai munkamegosztás és stack-indoklás}

A Python kiváló az adatgyűjtéshez, a JSON-feldolgozáshoz és a hálózatépítés gyors iterációjához. A Rust sokkal alkalmasabb a nagyméretű, ismételten meghívott gráfalgoritmusokhoz, ahol fontos a determinisztikus viselkedés, a teljesítmény és a strukturált export. Az SQLite kis projektméret mellett is kényelmesen biztosítja a perzisztenciát, a React pedig jó eszköz a demo- és elemzőfelületek építéséhez. A Python és Rust közti munkamegosztás éles: Python megtartja a gyors adatmanipuláció előnyeit, Rust a stabil futásidejű gráfmagot kapta. A kettő közé a Node/Express réteg került, amely szerződéses API-határként működik.

Az alábbiakban rétegenkénti bontásban szerepel, miért lett az adott technológia a végső választás, milyen alternatívák merültek fel, és mik a főbb hátrányok.

\textbf{Python (adatgyűjtés, szkriptréteg).} Gazdag ökoszisztéma (Riot API kliens, NetworkX, pandas), rövid iterációs ciklus, könnyen módosítható gyűjtési logika. Alternatíva: Go vagy Ruby szkriptekkel gyorsabb indulás lett volna lehetséges, de a tudományos ökoszisztéma (NetworkX, scipy) Pythonhoz kötött. Hátránya: gyengébb teljesítmény nagy gráfműveleteknél -- ezt a Rust réteg váltja ki.

\textbf{SQLite (perzisztencia).} Szervermentes, önálló fájl, könnyű verziókezelés, a Python és a Rust egyaránt natívan kezeli. Alternatíva: PostgreSQL erősebb párhuzamos írást és komplex lekérdezéseket adott volna -- de a projekt egyszerre egy felhasználós, lokálisan futtatható rendszer. A SQLite kompromisszuma ennél a hatókörnél kedvezőbb \cite{sqlite}. Hátránya: egyidejű írás nincs, és a séma módosítása migrációt igényel.

\textbf{Node.js / Express (API-héj).} A frontend (React) és a backend ugyanazon ökoszisztémán belül él; az Express könnyen proxyzza a Rust-binárist és a Python-szkripteket. Alternatíva: FastAPI (Python) logikusabb lett volna a Python-réteg mellé, de Node jobb aszinkron I/O-t adott a Riot API relay-ekhez, és a React/Vite dev-flow integrációja egyszerűbb lett. Hátránya: CPU-intenzív számítások nem a Node-ban futnak.

\textbf{Rust (gráfanalitika, útvonalkeresés).} Nulla futásidejű overhead, biztonságos memóriakezelés, Rayon párhuzamosítás, egzakt determinizmus \cite{jung2020,rustlang}. Alternatíva: C++ ugyanolyan teljesítményt adna, de a Rust borrow checker megelőzi az érvénytelen iterátor-hibákat, amelyek gráf-traversal kódban gyakoriak. Hátránya: meredekebb tanulási görbe, hosszabb fordítási idő.

\textbf{React / Vite (frontend).} Komponensalapú UI, nagy ökoszisztéma (MUI, Three.js), gyors hot reload. Alternatíva: Vue vagy Svelte kisebb bundle-t adna, de a MUI és a Three.js React-integráció érettebb. Hátránya: overkill lenne egy egyoldalas demóhoz; a projekt több vizualizációs oldalt igényel, ami igazolja a frameworköt.

\subsection{A C\# böngésző különállása}

A C\# / ASP.NET Core adatbázis-böngésző különálló eszközként él, nem a React frontend részeként. A fő frontend kutatási és vizualizációs célú (gráfnézetek, pathfinder, analitikák), a böngésző adatbázis-inspekciós eszköz (nyers táblasorok, JSON-értékek, szűrés, lapozás). Ezek különböző felhasználói igényeket elégítenek ki, és különböző stack-et igényelnek. Az ASP.NET Core-os megközelítés JavaScriptes kliensoldali logika nélkül is stabil, könnyen telepíthető eszközt ad.

\subsection{A Node/Express réteg szerepe}

A \textit{backend/server.js} alapján a Node rétegnek önálló felelőssége van: dataset-regisztert kezel és aktív datasetet választ, futás közbeni kulcsokat és környezeti útvonalakat központosít, háttércollector folyamatokat indít és monitoroz, HTTP-végpontokat biztosít a Rust parancsok fölé, és a bird's-eye artifaktumokat és egyéb cache-eket konzisztensen szolgálja ki. Ez a köztes réteg csökkenti a frontend és a számítási mag közti csatolást. A böngészőnek nem kell folyamatindítással, fájlrendszer-útvonalakkal vagy dataset-propagálással foglalkoznia.

A regiszterben négy dataset szerepel. A \textit{default} az eredeti, vegyes queue-tartalmú alapkorpusz, amely a korai fejlesztési előtörténetet őrzi, kutatási célra nem alkalmas. \subsection{A dataset mint életciklus-egység}

A dataset a jelenlegi rendszerben nem statikus mappát, hanem együtt mozgó állapotcsomagot jelent. Egy új dataset létrehozása után több, egymástól elválasztható fázison mehet keresztül:

\begin{enumerate}
    \item nyers seed és játékosállapot kialakítása;
    \item match collector futtatása;
    \item PUUID-alapú normalizálás és score-frissítés;
    \item legacy Python export vagy populációs klaszterperzisztencia, ha az adott futás ezt igényli;
    \item Rust runtime-felépítés, Graph V2 artifact-export és bird's-eye export;
    \item kutatási analitikák lefuttatása.
\end{enumerate}

Ez a bontás azért erős, mert világos köztes állapotokat ad. Egy asszortativitási vagy központisági futtatásnak nem kell újra elindítania a Riot API-gyűjtést, és egy klaszterezési változtatásnak sem kell a teljes pipeline-t újraépítenie.

\subsection{A dataset-választás propagálása}

A dataset-választás nem áll meg a Node-szinten. A backend a kiválasztott datasetből környezeti változókat képez a Rust futtatókörnyezet felé (\textit{DATASET\_ID}, \textit{GRAPH\_DB\_PATH}, \textit{PATHFINDER\_MATCH\_DIR}, \textit{PATHFINDER\_BIRDSEYE\_CACHE\_DIR}), amelyeket a Rust runtime fogyaszt. Nincs külön "frontend igazság" és "Rust igazság" ugyanarról az adatállapotról: a dataset-választás a teljes végrehajtási láncon végigmegy, és a frontend által mutatott eredmény ugyanabból a datasetből jön, amelyen a kutatási analitika fut.

\section{A jelenlegi adatállapot}

Az aktív adatállapot 2026. április 28-án közvetlenül a repository aktuális állapotából lett ellenőrizve. Az aktív dataset a \textit{soloq}. Az ellenőrzött fő adatállományok és adatbázisok adatai:

\textbf{Flexset dataset (EUNE Master+ Flex Queue):}
\begin{itemize}
    \item meccs-JSON állományok: \textbf{4981} fájl a \textit{backend/data/matches/flexset/} könyvtárban;
    \item Graph V2 összegző jelentés: \textit{matchCount: 4980} (egy fájl eltérés ellenőrizendő);
    \item nyers adatbázis (\textit{players.db}): \textbf{15\,147} játékos;
    \item finomított adatbázis (\textit{playersrefined.db}): \textbf{15\,421} játékos, \textbf{599} klaszter, \textbf{5741} klasztertag.
\end{itemize}

\textbf{SoloQ dataset (EUNE Master+ Solo/Duo Queue, aktív):}
\begin{itemize}
    \item meccs-JSON állományok: \textbf{2001} fájl a \textit{backend/data/matches/soloq/} könyvtárban;
    \item Graph V2 összegző jelentés: \textit{matchCount: 2000};
    \item nyers adatbázis (\textit{players.db}): \textbf{2822} játékos;
    \item finomított adatbázis (\textit{playersrefined.db}): \textbf{6768} játékos, \textbf{140} klaszter, \textbf{1538} klasztertag.
\end{itemize}

A soloq dataset klaszterei jellemzően \emph{duo párokat} jelölnek: mivel a Solo/Duo Queue maximálisan kétfős premade csapatokat enged, az ismétlődő kapcsolatokból kiemelkedő klaszterek döntő többsége két-három játékosra szűkül. Ez strukturálisan különbözik a flexset klasztereitől, ahol akár ötfős premade csapat is kialakulhat rendszeresen ismétlődő együttsként.

A jelen dolgozat nem hipotetikus rendszert ír le, hanem egy olyan állapotot, amelyben a pipeline két, egymással összehasonlítható, eltérő queue-szemantikájú, valós adatokkal feltöltött és ténylegesen lefuttatott kutatási korpusszal rendelkezik.

A meccsszámban megfigyelhető egy-fájlos eltérés (4981 fájl vs. 4980 matchCount a Graph V2 összegzőben) metodikai megjegyzést igényel: ez feltehetően egy olyan meccsfájl, amelyet a gyűjtő letöltött, de a Graph V2 pipeline valamilyen validációs szűrőn kizárt. Ez nem kritikus eltérés, de jól mutatja, hogy a pipeline nem pusztán fájlszámlálás alapján dolgozik.

\subsection{Az SQLite konfiguráció}

Az SQLite ebben a projektben nem egyszerűségi választás, hanem jó kompromisszum: nem igényel külön adatbázisszerver-felügyeletet, a séma könnyen verziózható, lokálisan gyors, és demóhoz sem kell külön infrastruktúra. A normalizáló pipeline nem alapbeállításokkal használja: \textit{WAL} naplózást, \textit{NORMAL} szinkronizációt, memóriabeli ideiglenes tárolást és foreign key támogatást állít be. Ez lokálisan futó, ismételten újraíró, de nem többszáz egyidejű felhasználót kiszolgáló analitikai adattárhoz jó arány sebesség, stabilitás és hordozhatóság között.

\subsection{Klasztermodell}

A klaszterek perzisztenciáját a \textit{clusters} és \textit{cluster\_members} páros adja. A \textit{clusters} tábla főbb elemei: \textit{cluster\_id}, \textit{cluster\_type}, \textit{algorithm}, \textit{size}, \textit{summary\_json}, térbeli koordináták, build-verzió és frissítési idő. A \textit{cluster\_members} tábla a tagsági és szerepkörjellegű információt tárolja: klaszter-játékos összerendelést, \textit{is\_bridge}, \textit{is\_star}, \textit{is\_best\_op}, \textit{is\_worst\_feed} jelzőket, és szerepköri összegzést \textit{role\_json} formában.

A \textit{backend/cluster\_persistence.py} a klasztereket \textit{cluster\_type} szerint cseréli: egy új \textit{python\_population} build felülírja a régi Python-klasztereket, de nem nyúl a \textit{rust\_pathfinding} családhoz. Így ugyanazon adatbázisban több klasztercsalád él együtt anélkül, hogy egy újrafuttatás mindent elsöpörne. A \textit{summary\_json}, \textit{role\_json}, \textit{build\_version} és \textit{updated\_at} mezők a kutatási visszakövethetőséget is javítják.

\subsection{A két klasztercsalád}

A rendszerben két klasztercsalád él párhuzamosan. A \textit{python\_population} típusú klaszterek a Python gráfépítési pipeline kimenetei: NetworkX-alapú modularitás-maximalizálás, kemény méretkorlát nélkül, közösségdetektálási perspektívával. A \textit{rust\_pathfinding} típusú klaszterek a Rust motor futásidejű klaszterezéséből jönnek: ally-kapcsolatokon alapuló erős-összefüggési szegmensek, maximum 20 tagos korláttal, pathfinding és vizualizációs célra optimalizálva. Az \textit{orbitScore} és \textit{orbitRadius} értékek kizárólag a Rust klaszterekre értelmezhetők.

A két família egymást kiegészíti: a Python klaszterek adnak egy moduláris alapú közösségstruktúrát, a Rust klaszterek ennek futásidejű, vizualizáció-kompatibilis változatát. A klaszterprofilok aggregált teljesítménymutatói (\textit{opscore} és részmetrikák klaszterszinten) egyben a Tribal NeuroSim szimulációjának genomszekvencia-bemenetét is alkotják: minden szimulált törzs egy ilyen adatbázis-klaszter profilja alapján indul.

A két kutatási dataset klaszterstruktúrája lényegesen eltér. A flexset finomított adatbázisában \textbf{599} klaszter és \textbf{5741} tagsági rekord van, a Graph V2 artifaktumban \textbf{1531} látható klaszter, átlagos méret \textbf{3.750}, legnagyobb \textbf{20}. A soloq-ban \textbf{140} klaszter és \textbf{1538} tagsági rekord van, a Graph V2 artifaktumban \textbf{870} látható klaszter, átlagos méret \textbf{1.768}, legnagyobb \textbf{20}. A flexset átlagos klasztermérete közel duplája a soloq-énak, ami empirikusan is alátámasztja, hogy a Flex Queue erősebb ismétlődő csoportkapcsolatokat termel.
