\chapter{A rendszer felépítése és evolúciója}

\section{A korai változat}

Az eredeti rendszer jóval közelebb állt egy adatvezérelt játékosnézőhöz, mint a jelenlegi kutatási platform. A korai pontozási logika (\textit{add\_new\_players.js}) egyszerű képletekkel dolgozott. A korai \textit{build\_graph.py} Python-központú volt: meccsekből épített kapcsolati gráfot, és HTML- és PyVis-exportokat készített. Ezen az úton váltak végül a klaszterprofilok a Tribal NeuroSim szimulációs bemenetének alapjává.

\section{A jelenlegi rendszer}

A rendszer több egymástól jól elkülönített rétegre épül. Komponensei: Python (adatgyűjtés, gráfépítés, legacy-exportok), SQLite (központi perzisztencia), Node.js/Express (API-héj és orchesztráció), Rust (útvonalkeresés, gráfanalitika, globális artifact-export, szimuláció logika), React/Vite (frontend és vizualizáció), C\# ASP.NET Core (különálló adatbázis-böngésző) és a C\# MonoGame (Tribal NeuroSim vizualizációs rétege, megjeleníti a Rust payloadokat).

A komponensdiagram a kibővített verzió 5. fejezetében található \cite{premadegraph-full}; a lite verzióban az általános leírást a szöveg tartalmazza.

\section{A technológiai stack indoklása}

Minden technológiai döntéshez volt alternatíva; az alábbiakban az egyes rétegek kiválasztásának főbb szempontjai, valamint a mérlegelt versenytársak szerepelnek.

\textbf{Python (adatgyűjtés, szkriptréteg).} Gazdag ökoszisztéma (Riot API-kliens, NetworkX, pandas), rövid iterációs ciklus, könnyen módosítható gyűjtési logika. Alternatíva: Go vagy Ruby szkriptekkel gyorsabb indulás lett volna lehetséges, de a tudományos ökoszisztéma (NetworkX, scipy) Pythonhoz kötött. Hátránya: gyengébb teljesítmény nagy gráfműveleteknél, ezt a Rust-réteg helyettesíti.

\textbf{SQLite (perzisztencia).} Szervermentes, önálló fájl, könnyű verziókezelés: a Python és a Rust egyaránt natívan támogatja. Alternatíva: PostgreSQL erősebb párhuzamos írást és komplex lekérdezéseket adott volna, de a projekt egyszerre egy felhasználós, lokálisan futtatható, bemutatható rendszer. A SQLite kompromisszuma ennél a hatókörnél kedvezőbb \cite{sqlite}. Hátránya: egyidejű írás nincs, és a séma módosítása migrációt igényel.

\textbf{Node.js / Express (API-héj).} A frontend (React) és a backend ugyanazon ökoszisztémán belül élnek; az Express könnyen proxyzza a Rust-binárist és a Python-szkripteket. Alternatíva: FastAPI (Python) logikusabb lett volna a Python-réteg mellé, de mivel a Python-réteg le lett fokozva kizárolag adatgyűjtésre és egyszerűbb összesitésekre , ezért a Node jobb aszinkron I/O-t adott a Riot API relay-ekhez, és a Vite integrációja egyszerűbb lett. Hátránya: CPU-intenzív számítások nem a Node-ban futnak.

\textbf{Rust (gráfanalitika, útvonalkeresés).} Nulla futásidejű overhead, biztonságos memóriakezelés, Rayon párhuzamosítás, egzakt determinizmus \cite{jung2020,rustlang}. Alternatíva: C++ ugyanolyan teljesítményt adna, de a Rust borrow checker megelőzi az érvénytelen iterátor-hibákat, amelyek gráf-traversal kódban gyakoriak. Go könnyebb concurrency-modellt adna, de gyengébb cache-teljesítménnyel. Hátránya: meredekebb tanulási görbe, kínzó fordítás élmény, hosszabb fordítási idő.

\textbf{React / Vite (frontend).} Komponensalapú UI, nagy ökoszisztéma (MUI, Three.js), gyors hot reload. Alternatíva: Vue vagy Svelte kisebb bundle-t adna, de a MUI és a Three.js React-integrációja érettebb. Hátránya: overkill lenne egy egyoldalas demóhoz; a projekt több vizualizációs oldalt igényel, ami igazolja a frameworköt.

\section{Több dataset kezelése}

Az implementáció több, egymástól független adatkészletet kezel. A \textit{backend/data/datasets.json} regiszter alapján minden dataset külön rendelkezik nyers adatbázissal, finomított adatbázissal, meccs-JSON-könyvtárral és cache-könyvtárral.

Az adatkészlet-választás nem áll meg a Node-szinten. A backend a kiválasztott datasetből környezeti változókat képez a Rust futtatókörnyezet felé (\textit{DATASET\_ID}, \textit{GRAPH\_DB\_PATH}, \textit{PATHFINDER\_MATCH\_DIR}, \textit{PATHFINDER\_BIRDSEYE\_CACHE\_DIR}).

A rendszerben két klasztercsalád működik párhuzamosan. A \textit{python\_population} típusú klaszterek a Python-gráfépítési pipeline kimenetei: NetworkX-alapú modularitásmaximalizálás, közösségdetektálási perspektívával. A \textit{rust\_pathfinding} típusú klaszterek a Rust motor futásidejű klaszterezéséből jönnek: ally-kapcsolatokon alapuló, erős összefüggési szegmensek, maximum 20 taggal. A klaszterprofilok aggregált teljesítménymutatói egyben a Tribal NeuroSim szimulációjának genomszekvencia-bemenetét is képezik.
