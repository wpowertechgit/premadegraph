## Purpose

This document describes a future research-oriented integration between `premadegraph` and the separate `genetic-neurosim` project.

It is intentionally planning-only.

The purpose of the integration is not to claim an objective ranking of all players, and not to replace rank, MMR, or matchmaking systems. The purpose is to create an exploratory simulation framework in which player clusters derived from real match and graph data can be compared under a shared evolutionary environment.

## In Plain Language

This document is a future-work idea, not a description of something already built.

In simple terms, it asks: if we take the player groups discovered by this project and feed them into a separate simulation system, what kinds of long-term patterns might appear? It is about exploration and research framing, not about declaring "who is best."

## Core Framing

The integration should be described using the following principles:

- We derived population-level simulation inputs from observed graph and match-performance features.
- We compared the evolutionary robustness of cluster-defined player groups under a shared simulation environment.
- The simulation is exploratory and assumption-dependent.
- Results are interpreted as model behavior, not direct proof of real-world superiority.
- Different cluster profiles exhibit different long-term survival and adaptation patterns.

These statements are the academic core of the integration and should remain stable even if the technical implementation changes.

## Why This Integration Makes Sense

`premadegraph` already produces structured information about player relationships, clusters, cohesion, performance summaries, temporal stability, and potentially higher-order graph metrics.

`genetic-neurosim` already provides a simulation framework with:

- populations
- generations
- mutation
- repeated competition
- replay and inspection
- configurable environmental parameters

Because of that, the two projects are compatible at a conceptual level:

- `premadegraph` supplies observed cluster-level structure and performance summaries
- `genetic-neurosim` supplies the dynamic sandbox in which those cluster-derived profiles can be stress-tested

This creates a useful experimental bridge between static graph analysis and dynamic evolutionary behavior.

## High-Level Research Question

If player communities are transformed into population-level simulation profiles, which kinds of cluster structures remain robust, dominant, or persistent under repeated competitive selection?

This is intentionally narrower and more defensible than asking who the objectively best players are.

## Unit Of Analysis

The default integration model should be:

- one graph cluster equals one simulation population
- multiple clusters coexist in the same simulation sandbox
- all populations are subjected to the same simulation rules

This is preferable to using isolated single-cluster simulations, because the comparative environment is part of the experiment.

## Data Source From Premadegraph

The simulation should consume only features that can be derived from the match dataset and graph structure.

Preferred inputs include:

- cluster size
- average opscore
- average feedscore
- opscore variance
- feedscore variance
- cluster density or cohesion
- average internal edge strength
- internal versus external edge ratio
- temporal stability metrics
- assortativity summaries
- centrality summaries
- signed-balance summaries if available later

These values should be exported as structured cluster profiles rather than as ad hoc labels.

## Recommended Data Contract

The cleanest interface is a cluster-profile export generated by `premadegraph` and consumed by `genetic-neurosim`.

A future export format should contain, at minimum:

- cluster identifier
- cluster size
- graph summary metrics
- performance summary metrics
- stability summary metrics
- optional confidence or data coverage indicators

The export should remain descriptive, not interpretive.

For example, the contract should prefer fields such as:

- `mean_opscore`
- `feedscore_stddev`
- `cohesion`
- `internal_edge_ratio`

Instead of speculative fields such as:

- `aggressive`
- `toxic`
- `smart`

Interpretive parameters should be derived downstream inside the simulation layer, where the mapping is explicit and documented.

## Mapping Strategy

The recommended first approach is genome-proxy seeding.

That means cluster metrics from `premadegraph` are converted into normalized simulation parameters that influence the initial state of a population.

Examples of safe mappings:

- high average `opscore` may increase baseline effectiveness
- low performance variance may increase stability
- strong cohesion may improve coordination or resistance to fragmentation
- high internal edge ratio may support persistence under pressure
- strong signed imbalance, if used later, may increase volatility

These mappings should be treated as simulation assumptions, not discovered truths.

## Simulation Model

The intended setup is a shared arena in which several cluster-derived populations are introduced at once.

The simulation then observes:

- which populations dominate early
- which populations adapt across generations
- which populations collapse quickly
- which populations remain stable over long horizons
- whether compact elite clusters outperform large resilient communities

This shared-sandbox approach is the main scientific value of the integration.

## Apex Cluster Scenario

A specific experimental scenario worth documenting is the contrast between:

- small elite clusters with very high average performance
- larger clusters with lower peak performance but stronger resilience and redundancy

In informal language, the first type may look like an apex predator population.

In formal language, these should be described as:

- elite low-cardinality clusters
- compact high-performance populations
- high-intensity small communities

The key experiment is not simply whether they win quickly, but whether they remain viable over time when placed in competition with larger, more stable populations.

This distinction is important because it opens a deeper question:

- is short-term dominance more important than long-term resilience?

That is a strong comparative research angle.

## Expected Outputs

A meaningful future integration should produce results such as:

- survival duration per population
- generation-to-generation stability
- dominance share over time
- collapse frequency
- adaptation rate under mutation
- replayable simulation traces
- comparisons between initial profile strength and long-term persistence

The most useful outputs will compare populations against one another rather than merely producing one final winner.

## Interpretation Rules

The integration should respect the following boundaries:

Valid interpretations:

- certain cluster-derived profiles appear more robust under the chosen simulation rules
- some population structures dominate quickly but fail to persist
- some larger communities show stronger long-term survival despite lower peak performance
- simulation outcomes are sensitive to cluster structure and performance-derived priors

Invalid or overreaching interpretations:

- this proves the objectively best players
- this replaces MMR or rank
- this reveals the true strongest real-world community
- this predicts real future match outcomes without further validation

Keeping these boundaries explicit is essential for thesis safety and scientific honesty.

## Academic Contribution

This integration would add value to the broader research story in several ways:

- it extends the project from static network analysis into dynamic simulation
- it turns graph-derived communities into experimentally testable population models
- it provides a novel way to compare high-performance and high-resilience groups
- it strengthens the systems side by linking two substantial technical projects
- it creates a clean separation between observed data, modeling assumptions, and simulation outcomes

This is academically stronger than making broad predictive claims, because it preserves interpretability.

## Suggested Work Sequence

When this integration is eventually attempted, the recommended order is:

1. finalize which cluster-level metrics will be exported from `premadegraph`
2. define a stable schema for cluster profiles
3. normalize those metrics into simulation-ready ranges
4. document the metric-to-parameter mapping assumptions
5. create a first shared-sandbox scenario with several populations present at once
6. compare compact elite clusters against larger resilient ones
7. evaluate outcome stability across repeated runs
8. only then add richer visual or interactive presentation layers

## Non-Goals

The integration is not meant to do the following:

- replace the core graph-analysis thesis work
- declare an absolute real-world ranking of players
- hide modeling assumptions behind flashy simulation output
- treat simulation outcomes as direct factual proof about player skill
- become so complex that the mapping between data and behavior is no longer interpretable

## Final Note

This integration is most valuable when treated as an exploratory simulation layer built on top of real cluster and match-performance data.

Its strength comes from disciplined framing:

- observed structure from `premadegraph`
- explicit assumptions in the mapping layer
- evolutionary comparison inside `genetic-neurosim`
- cautious interpretation of outcomes

If those boundaries are respected, the combined direction can become both technically impressive and academically meaningful.
